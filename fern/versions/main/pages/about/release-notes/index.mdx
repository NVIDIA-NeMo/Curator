# NeMo Curator Release Notes: 26.06 (v1.3.0)

NeMo Curator 26.06 expands text, audio, video, and synthetic-data curation, and makes pipeline resource configuration more consistent across execution backends.

<Warning>
**Python 3.11 or later is required.** NeMo Curator 26.06 supports Python 3.11 through 3.13. Before upgrading, review the [26.06 migration checklist](/about/release-notes/26-06-migration), especially the framework-owned task IDs, sentinel task construction, removed video preprocessing option, audio ASR worker rename, and architecture-specific dependency changes.
</Warning>

## What's New

### Nemotron-CLIMB Data Curation

The new [Nemotron-CLIMB tutorial](/curate-text/tutorials/nemotron-climb) implements the data-mixture optimization workflow introduced in [#1727](https://github.com/NVIDIA-NeMo/Curator/pull/1727) and completed in [#2072](https://github.com/NVIDIA-NeMo/Curator/pull/2072). The workflow:

- Embeds and clusters candidate documents.
- Prunes and merges clusters with FastText classifiers.
- Produces Megatron-compatible `.bin` and `.idx` token files.
- Samples Dirichlet data mixtures, trains and evaluates proxy models, and fits a LightGBM predictor.
- Narrows the mixture search from 64 candidates to 32, 16, and a final selected mixture.

See [Nemotron-CLIMB stages](/curate-text/tutorials/nemotron-climb/stages) for the stage-by-stage API.

### Audio Tagging, Quality, and Punctuation-and-Capitalization Preparation

The [audio tagging tutorial](/curate-audio/tutorials/audio-tagging) added in [#1863](https://github.com/NVIDIA-NeMo/Curator/pull/1863) composes resampling, diarization, long-segment splitting, ASR alignment, bandwidth estimation, SQUIM quality metrics, and TTS or ASR manifest preparation. Optional second-pass ASR enables pairwise WER scoring. The pipeline can also apply inverse text normalization and Chinese text conversion when their optional dependencies are available.

New public stages include `BandwidthEstimationStage`, `TorchSquimQualityMetricsStage`, `ComputeWERStage`, `GetPairwiseWerStage`, and `PrepareModuleSegmentsStage`.

### Long-Form Audio Pretraining Pipeline

The [audio pretraining tutorial](/curate-audio/tutorials/audio-pretrain) from [#1898](https://github.com/NVIDIA-NeMo/Curator/pull/1898) turns diarized and transcribed long-form recordings into bounded, mono training snippets. It includes overlap filtering, snippet cut planning, repeated-content filtering, extraction, tar and JSONL writers, metrics, and a dry-run mode through `build_audio_pretrain_pipeline()`.

### Nemotron OCR Synthetic Data Generation

The [Nemotron OCR pipeline](/curate-text/synthetic/nemotron-ocr) added in [#1899](https://github.com/NVIDIA-NeMo/Curator/pull/1899) generates OCR training records from Hugging Face datasets with `OCRNemotronV2Stage`. An optional `OCRScoringQAStage` uses Nemotron Nano Omni to score generated records and create question-answer data. See the [OCR tutorial](/curate-text/synthetic/nemotron-ocr/tutorial) for an end-to-end example.

### Video Caption Quality Evaluation

The [caption evaluation workflow](/curate-video/process-data/caption-evaluation) from [#1980](https://github.com/NVIDIA-NeMo/Curator/pull/1980) builds a diverse benchmark from sampled OpenVid clips, summarizes captions to the model's token budget, and uses Cosmos-Embed1 cosine similarity to evaluate video-text alignment. Summaries are cached so repeated evaluations do not regenerate them.

<Note>
The removed `--captioning-model-does-preprocess` option appears in an older benchmark README. Do not use it with 26.06; follow the [video preprocessing migration](/about/release-notes/26-06-migration#remove-the-video-model-preprocessing-option).
</Note>

### Expanded Translation Controls

`TranslationStage` gained production-oriented controls in [#2038](https://github.com/NVIDIA-NeMo/Curator/pull/2038), including segmentation mode and minimum segment length, prompt and generation configuration, concurrency and health checks, FAITH quality scoring and filtering, output modes, message reconstruction, and already-translated record handling. See [Translation](/curate-text/process-data/language-management/translation) for the complete configuration surface.

### Nemotron-Parse Throughput Tuning and Metrics

Nemotron-Parse now forwards `engine_kwargs` such as `max_num_seqs` and `enforce_eager` to vLLM and reports per-task inference time, pages, prompt and output tokens and characters, truncation, empty results, and retry counts. PDF partitioning also fans work out across workers. These changes landed in [#2054](https://github.com/NVIDIA-NeMo/Curator/pull/2054); see [Parse PDFs with Nemotron-Parse](/curate-text/load-data/nemotron-parse-pdf).

### Consistent Stage Scheduling and `with_()` Overrides

Stage scheduling was standardized in [#2077](https://github.com/NVIDIA-NeMo/Curator/pull/2077) and [#2086](https://github.com/NVIDIA-NeMo/Curator/pull/2086):

- Use fixed `ActorPoolStrategy` or `TaskPoolStrategy` worker counts for Ray Data.
- Use minimum, maximum, and initial worker bounds for autoscaling actor pools.
- Describe Xenna resources per worker or per node.
- Apply per-instance scheduling overrides with `stage.with_(...)`.

See [Stage worker sizing](/reference/infra/stage-worker-sizing) before carrying custom worker settings forward.

## Improvements

### Deduplication Input Extension Discovery

Deduplication workflows now infer input extensions from `input_filetype` in [#2045](https://github.com/NVIDIA-NeMo/Curator/pull/2045): Parquet scans `.parquet`, while JSONL scans both `.jsonl` and `.json`. Explicit `input_file_extensions` still takes precedence, and unsupported file types fail early. See [Deduplication](/curate-text/process-data/deduplication).

### Reliable FLEURS Downloads

The FLEURS tutorial now uses the Hugging Face content cache and retry behavior through `hf_hub_download`, supports `cache_dir`, and lets benchmarks set `auto_download=False`. This reduces rate-limit failures and repeated downloads ([#2057](https://github.com/NVIDIA-NeMo/Curator/pull/2057)).

### Safer ArXiv Extraction

`ArxivIterator(extract_tmp_dir=None)` now extracts into the system temporary directory by default instead of the download directory. Set `extract_tmp_dir` to retain explicit placement ([#2055](https://github.com/NVIDIA-NeMo/Curator/pull/2055)).

### Dynamo Server Configuration

[Dynamo fixes in #2053](https://github.com/NVIDIA-NeMo/Curator/pull/2053) propagate `DynamoServerConfig.subprocess_env` to the frontend and workers, support explicit `dyn_chat_processor="vllm"` routing for pass-through multimodal requests, validate hybrid KV-cache settings, and render false-valued CLI options as `--no-*` flags.

### NeMo Data Designer Tutorial Guidance

The NDD tutorial now configures stage workers with `.with_(num_workers=2)` rather than a Xenna-only stage spec ([#2104](https://github.com/NVIDIA-NeMo/Curator/pull/2104)). It also documents a workaround for local multi-GPU `InferenceServer` startup hangs on PCIe systems: restart Ray or the kernel and either set `NCCL_P2P_DISABLE=1` or use `tensor_parallel_size=1` ([#2114](https://github.com/NVIDIA-NeMo/Curator/pull/2114)). See [NeMo Data Designer](/curate-text/synthetic/nemo-data-designer).

### Video CPU Stage Scheduling

`VideoFrameExtractionStage` and `ClipTranscodingStage` now request one Ray Data CPU so adjacent CPU stages can be fused more effectively ([#2110](https://github.com/NVIDIA-NeMo/Curator/pull/2110)). Their Xenna resource declarations remain larger because they describe worker capacity for FFmpeg work; the Ray reservation does not reduce the computation performed by the stage.

## Fixes

- **Full TCP port range:** Automatic port selection now includes port 65535 ([#2024](https://github.com/NVIDIA-NeMo/Curator/pull/2024)).

## Breaking Changes

The [26.06 migration checklist](/about/release-notes/26-06-migration) provides before-and-after examples for every item below.

- **Framework-owned task IDs:** `Task.task_id` is no longer accepted by task constructors. Executors assign deterministic IDs at stage boundaries ([#2036](https://github.com/NVIDIA-NeMo/Curator/pull/2036), [#2058](https://github.com/NVIDIA-NeMo/Curator/pull/2058)).
- **Sentinel task classes:** Construct empty sentinels with `EmptyTask()` and check them with `isinstance`. `SentinelTask` is the payloadless base class ([#2062](https://github.com/NVIDIA-NeMo/Curator/pull/2062), [#2070](https://github.com/NVIDIA-NeMo/Curator/pull/2070)).
- **Video preprocessing option removed:** Remove `model_does_preprocess` and `--captioning-model-does-preprocess`; video models now use vLLM or Hugging Face preprocessing ([#2109](https://github.com/NVIDIA-NeMo/Curator/pull/2109)).
- **Audio ASR worker rename:** Replace stage argument `num_workers` with `dataloader_num_workers`. `num_workers` is now reserved for execution-worker configuration through `with_()` ([#2086](https://github.com/NVIDIA-NeMo/Curator/pull/2086)).

## Dependency Changes

- **`nemo_text_processing` on Arm:** The audio extra installs `nemo_text_processing` on x86_64 Linux, but no longer installs it on aarch64 or macOS because its `pynini` dependency is not supported there. Audio stages that import it, including inverse text normalization and WER processing, must be omitted or supplied through a separately supported environment on those platforms ([#2049](https://github.com/NVIDIA-NeMo/Curator/pull/2049)).
- **`flash-attn` is no longer a direct video dependency:** The `video_cuda12` extra no longer pins or installs `flash-attn`. Built-in video pipelines use their declared dependencies; custom code that imports FlashAttention must install a compatible build explicitly ([#2107](https://github.com/NVIDIA-NeMo/Curator/pull/2107)). This applies to both x86_64 and aarch64.

See the [architecture dependency matrix](/about/release-notes/26-06-migration#review-architecture-specific-dependencies) for upgrade actions.

## Release Boundary: Pipeline Resumability

<Warning>
Source-level checkpoint and resume support is **not included in NeMo Curator 26.06**. [#2033](https://github.com/NVIDIA-NeMo/Curator/pull/2033) was merged into a feature branch rather than `main`; the 26.06 `Pipeline.run()` API does not accept `checkpoint_path`. Do not rely on LMDB task metadata or `NoneTask` and `FailedTask` sentinels described by that proposal. This note will be removed when the implementation lands on the release branch.
</Warning>

## Upgrade Checklist

1. Create a Python 3.11, 3.12, or 3.13 environment.
2. Apply every relevant code change in the [26.06 migration checklist](/about/release-notes/26-06-migration).
3. Reinstall the modality extras used by your pipeline and verify the architecture-specific dependency matrix.
4. Run a representative pipeline with a small input set and inspect generated task IDs, worker counts, and output manifests.
5. Remove any checkpoint/resume configuration written against #2033 until that API lands on `main`.
