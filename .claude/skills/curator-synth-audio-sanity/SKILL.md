# End-to-end audio pipeline sanity check with synthetic audio

Use this when the user wants to verify that an audio pipeline (YAML config or Python script) runs correctly from input to output, but does not have real audio data on hand — or wants to avoid using real data for a quick iteration.

The approach: synthesize a small batch of audio clips via `MagpieTTSStage`, then feed those clips directly into the downstream pipeline (everything after the initial-manifest stage). If the whole chain runs without errors and produces output entries, the pipeline is structurally sound.

## Step 1 — Identify the target pipeline

- If the user supplied a path, read it. If not, ask.
- Detect format: Hydra YAML (`stages:` / `processors:` list of `_target_` dicts) vs Python (`Pipeline(...).add_stage(...)` calls).
- Identify the **first real stage** — i.e., the stage that would normally consume `audio_filepath`. Typically this is stage 1 (after a `ManifestReader` or initial-manifest creator). Note what key it expects for the audio file (`audio_filepath` is standard; some stages use a custom key like `resampled_audio_filepath`).
- Note any `???` Hydra interpolations (required CLI args the user must supply at runtime) — list them so the user knows what else to pass.

## Step 2 — Choose sample transcripts

Use this fixed set of short, phonetically diverse English sentences unless the user supplies their own:

```
English (en):
  "The quick brown fox jumps over the lazy dog."
  "She sells seashells by the seashore."
  "How much wood would a woodchuck chuck if a woodchuck could chuck wood?"
  "Peter Piper picked a peck of pickled peppers."
  "Betty Botter bought some butter, but the butter was bitter."
```

Five entries is the right default: small enough to finish fast, diverse enough to exercise the pipeline. If they provide their own text, use those exactly.

## Step 3 — Create the synthetic manifest

Write the entries to `/tmp/synth_sanity_manifest.jsonl` (or a path the user specifies):

```python
import json, pathlib

entries = [
    {"text": "The quick brown fox jumps over the lazy dog.",                           "language": "en"},
    {"text": "She sells seashells by the seashore.",                                   "language": "en"},
    {"text": "How much wood would a woodchuck chuck if a woodchuck could chuck wood?", "language": "en"},
    {"text": "Peter Piper picked a peck of pickled peppers.",                          "language": "en"},
    {"text": "Betty Botter bought some butter, but the butter was bitter.",             "language": "en"},
]
out = pathlib.Path("/tmp/synth_sanity_manifest.jsonl")
out.write_text("\n".join(json.dumps(e) for e in entries))
print(f"Wrote {len(entries)} entries → {out}")
```

Run this with the Bash tool. Confirm the file was written.

## Step 4 — Build the TTS-prefixed pipeline script

Create a self-contained Python runner at `/tmp/synth_sanity_run.py`. **Do not modify the user's pipeline files.**

Key decisions:
- Set `MagpieTTSStage.generated_audio_filepath_key` to `"audio_filepath"` (the standard key most downstream stages expect). If the downstream pipeline uses a different key (e.g. `resampled_audio_filepath`), match it here instead.
- Create the generated audio output dir at `/tmp/synth_sanity_audio/` (create it if it doesn't exist).
- If the user's pipeline is a **Python script**, import and reuse its stages list directly — don't duplicate stage construction. Read the Python file, identify how stages are assembled, and write a harness that prepends the TTS stage.
- If the user's pipeline is a **YAML config**, instantiate it with Hydra and prepend the TTS stage programmatically, or convert the YAML stages list into a Python `Pipeline` call (simpler for a sanity script).

Template (adjust stage list for the user's pipeline):

```python
# /tmp/synth_sanity_run.py — AUTO-GENERATED by curator-synth-audio-sanity skill
import pathlib
from nemo_curator.pipeline import Pipeline
from nemo_curator.stages.audio.inference.tts.magpie import MagpieTTSStage
from nemo_curator.stages.audio.common import ManifestReader, ManifestWriterStage

# ---- TTS synthesis stage ---------------------------------------------------
GENERATED_AUDIO_DIR = "/tmp/synth_sanity_audio"
pathlib.Path(GENERATED_AUDIO_DIR).mkdir(parents=True, exist_ok=True)

tts_stage = MagpieTTSStage(
    generated_audio_dir=GENERATED_AUDIO_DIR,
    generated_audio_filepath_key="audio_filepath",   # match downstream expectation
)

# ---- Downstream pipeline stages (from user's config) -----------------------
# TODO: replace / extend with stages from the user's pipeline
downstream_stages = [
    # <Stage1>(...),
    # <Stage2>(...),
]

# ---- Assemble & run --------------------------------------------------------
pipeline = Pipeline(
    backend="xenna",      # or "ray_data" if user specified
    output_dir="/tmp/synth_sanity_output",
)
pipeline.add_stage(ManifestReader(manifest_path="/tmp/synth_sanity_manifest.jsonl"))
pipeline.add_stage(tts_stage)
for stage in downstream_stages:
    pipeline.add_stage(stage)
pipeline.add_stage(ManifestWriterStage(output_path="/tmp/synth_sanity_output/result.jsonl"))

result = pipeline.run()
print(f"Finished. Tasks processed: {result.num_tasks_processed}")
print(f"Output manifest: /tmp/synth_sanity_output/result.jsonl")
```

Fill in the `downstream_stages` list from the user's actual pipeline before writing the file.

If the downstream pipeline already ends with a manifest writer, keep it. If not, append one.

## Step 5 — Key compatibility check (before running)

Before executing, verify key flow:

1. `MagpieTTSStage` outputs: `text`, `language`, `audio_filepath` (because we set `generated_audio_filepath_key="audio_filepath"`).
2. First downstream stage `inputs()`: read the class file and confirm the required keys are present in that output set.
3. If there's a mismatch (e.g., downstream needs `audio_filepath` but TTS stage uses a different key), fix `generated_audio_filepath_key` in the script before running.

Apply the same `inputs()`/`outputs()` walk from `curator-audio-pipeline-check` for the rest of the chain. If any stage will fail due to a missing key, report it now and ask the user before running.

## Step 6 — Run

```bash
python /tmp/synth_sanity_run.py 2>&1 | tee /tmp/synth_sanity_run.log
```

Use the Bash tool with a generous timeout (600 000 ms / 10 min — TTS model download + inference can take a few minutes on first run).

Watch for:
- `MagpieTTSModel.from_pretrained(...)` download — this can take 1–3 minutes on first call. Normal.
- CUDA OOM — if the user's GPU is small, the TTS model + downstream model may not co-exist. Suggest running stages sequentially (TTS first, write intermediate manifest, then downstream pipeline from that manifest) if this happens.
- Any `ValueError` or `KeyError` from a stage — note the stage name and the missing key. Cross-check against Step 5.

## Step 7 — Verify output

After the run:

```bash
# Count output entries
wc -l /tmp/synth_sanity_output/result.jsonl

# Spot-check first entry
head -1 /tmp/synth_sanity_output/result.jsonl | python -m json.tool
```

Check:
- Output entry count matches (or is a reasonable subset of) the 5 input entries. If fewer, a filter stage dropped some — note which field triggered the filter.
- The keys expected by the user's pipeline end-stage are present in the output.
- `audio_filepath` (or equivalent) points to a file that was actually written to `/tmp/synth_sanity_audio/`.

## Step 8 — Report


## Handling YAML pipelines (detailed)

When the user's pipeline is a Hydra YAML:

1. Read the YAML. Identify the `stages:` (or `processors:`) key.
2. Skip the first stage if it is an initial-manifest creator (e.g. `CreateInitialManifestFleursStage`, `ALMManifestReader`) — the synthetic manifest replaces it.
3. For each remaining stage entry, resolve `_target_` to its Python class (same resolution as `curator-audio-pipeline-check` Step 3). Instantiate it in the harness with the YAML args passed as constructor kwargs.
4. Hydra interpolations like `${raw_data_dir}` — ask the user for their value or substitute a sensible `/tmp` default for the sanity run.
5. GPU resource declarations in the YAML — preserve them in the Python harness via `.with_(resources=..., batch_size=...)`.

## Handling two-stage GPU memory constraint

If the user's pipeline includes a large GPU model (e.g., a 1B+ param ASR model) **and** MagpieTTS (357 M params), they may not both fit on a single GPU at once. If you see an OOM error:

1. Run the TTS-only sub-pipeline first to produce an intermediate manifest with `audio_filepath` populated.
2. Write the intermediate manifest to `/tmp/synth_intermediate_manifest.jsonl`.
3. Run the downstream pipeline starting from that manifest (swap `ManifestReader` to point at the intermediate file, skip TTS stage).
4. Report this two-pass approach to the user and offer to automate it.

## Don't do this

- Don't modify the user's pipeline YAML or Python files — all changes go into the generated `/tmp/synth_sanity_run.py` only.
- Don't run on the user's real production data or manifests — only the synthetic manifest created in Step 3.
- Don't assume the downstream pipeline's first stage reads from `audio_filepath` — always check its `inputs()` declaration.
- Don't skip the key compatibility check (Step 5) — a key mismatch will silently drop all tasks when input validation is strict, and the run will appear to succeed with 0 output entries.
- Don't leave out the `ManifestWriterStage` at the end — without it there's no artifact to inspect.
- Don't report success if output entry count is 0 — that almost always means a key mismatch or an overly aggressive filter eliminated all synthetic entries.
- Don't hardcode `generated_audio_filepath_key="audio_filepath"` without checking — some pipelines use `resampled_audio_filepath` or another custom key.
